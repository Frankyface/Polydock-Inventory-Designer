# Handoff — PolyDock Inventory & Design Tool
_Last updated: 2026-07-06 · Current stage: Stage 4 — Stock-Aware BOM (done) · Stage 5 up next_

## 🎯 Goals
Give staff a 2D canvas to lay out real PolyDock modules/gangways/accessories at true scale, auto-derive the full parts list (BOM) from what's actually on the canvas, check it live against real inventory, and let a user commit a design so it deducts those parts from stock for real. This is "the disconnect" fix — the founding problem behind the whole project — and it's now done. Stage 5 (deploy & polish) is the last planned stage.

## 📍 Current State
Stages 1–4 are all functionally complete and verified in-browser via a real production-config build (not just the dev server). The app has: a tabbed inventory UI with CSV bulk import (Stages 1–2); a full design canvas (Stage 3) where staff create/list/edit designs, drag real dock modules/gangways onto an SVG grid at true scale, snap them flush, rotate in 90° steps, place freely-positioned accessories (cleats, bumpers, ladders, hinges), and set a background photo to align a design to a real property; and now (Stage 4) a live Bill of Materials panel under the canvas that auto-counts every placed module/gangway/accessory plus the connector or gangway-hinge hardware each seam implies, shows a per-part stock status ("OK" / "Short by N") against real inventory, and lets a user Commit a design (deducting the BOM from inventory) or Release it (restocking). A committed design freezes all edits until released. Still no backend — everything is client-side localStorage.

## 📂 Files I'm Working On
- `src/features/design/bom.js` — new: `computeBom(footprintItems, markerItems, seams, partsById)`, the pure function deriving the parts list from canvas state. Counts every placed item once, plus connector/gangway-hinge hardware from `computeSeams`' output; seams needing a part not in the catalog are counted in `unresolvedCount`, never silently dropped or guessed.
- `src/features/design/BomPanel.jsx` — new: presentational-only BOM table + stock status + Commit/Release buttons.
- `src/features/design/DesignEditor.jsx` — now the single place that computes `footprintItems`/`markerItems`/`seams`/BOM/stock (moved up from `DesignCanvas.jsx`, which previously computed these internally); owns `isLocked` (read live from `design.isCommitted`, not copied into state, so it stays reactive across a commit/release with no remount) and a ref-based `isSubmitting` guard so a rapid double-click on Commit/Release can't fire the action twice before the prop updates.
- `src/features/design/DesignCanvas.jsx` — simplified to accept `footprintItems`/`markerItems`/`seams` as props instead of computing them from a raw `items` array; `isLocked` now also blocks the background-photo drag layer, not just item drag.
- `src/features/design/DesignsPage.jsx` — `handleCommit`/`handleRelease`, using `applyBulkDeltas` (relative adjustments) instead of computing an absolute quantity from a stale render snapshot.
- `src/features/inventory/useLocalInventory.js` — new `applyBulkDeltas(deltas)`: reads the current quantity from *inside* the `setQuantities` state updater, not a value the caller precomputed — this is what makes commit/release correct even under a rapid double-click. `parts` is now memoized so its reference is stable across unrelated re-renders (downstream BOM/stock memoization was otherwise silently defeated). Also now returns `saveError`, surfaced as a top-level banner in `App.jsx`.
- `src/data/partsCatalog.js` — added `GANGWAY_ATTACHMENT_RULES`, mapping each gangway/ramp part to the hinge/connector accessory its attachment seam needs.
- `src/features/design/useDesigns.js` — `createDesign` now whitelists `isCommitted`/`committedBom` too (matching the lesson already learned once with `backgroundImage`).

## ✅ Things I've Changed
- Built the full Stage 4 stock-aware BOM: auto-computed parts list (modules + gangways + accessories + connectors + gangway hardware), live per-part stock check, and a commit/release flow that deducts/restocks inventory and freezes a committed design's canvas until released.
- Ran an 8-angle parallel code review (all 8 agents completed; findings cross-corroborated across multiple independent angles rather than needing a separate verify pass) and fixed everything that survived:
  - **Real bug, most significant (confirmed independently by 4 of 8 review angles):** `handleCommit`/`handleRelease` computed each part's new inventory quantity as an absolute value derived from a render-time snapshot (`inventoryByPartId[partId].quantityOnHand ± quantity`), not from the actual current state at write time — a real race under a rapid double-click or two commits in close succession. Fixed by adding `applyBulkDeltas` to `useLocalInventory.js`, which computes the new quantity *inside* React's state updater (always fresh), and switching `DesignsPage.jsx`'s commit/release to pass relative deltas instead of precomputed absolutes. Verified live by dispatching two synchronous clicks on Commit (and separately on Release) in the same tick and confirming inventory only moved once each time — the harder version of the real-world double-click case.
  - **Real bug:** even with delta-based updates, two clicks in the same tick could still fire `onCommit`/`onRelease` twice (each individually correct, but not idempotent together). Fixed with a ref-based `isSubmitting` guard in `DesignEditor.jsx` (a ref, not state, so the second click sees it set even before a re-render lands).
  - **Real gap:** background-photo controls (upload, reposition, scale, opacity, remove) had no `isLocked` guard, so a committed design's background could still be edited even though the "Committed — release to edit" badge implied everything was frozen. Fixed by locking all background controls too, for one unambiguous "committed = fully frozen" model, and by adding the missing `isLocked` guard to `handleSave` and the background-drag pointer handler in `DesignCanvas.jsx` for defense-in-depth.
  - **Real, currently-reachable performance issue:** `useLocalInventory`'s `parts` array was rebuilt (new reference) on every render, silently defeating the `inventoryByPartId` memoization in both `DesignsPage.jsx` and `DesignEditor.jsx` that keyed off it. Fixed by memoizing `parts` itself against `quantities`. This also let the now-unnecessary duplicate `inventoryByPartId` computation in `DesignsPage.jsx` be deleted entirely (a side effect of switching to delta-based updates, which don't need to read current quantities at the call site at all).
  - **Readability:** `commitBlockedReason`'s 4-branch nested ternary was flagged by 3 independent review angles as easy to misread. Replaced with an explicit early-return function (`getCommitBlockedReason`).
  - **Robustness gap:** no visibility if an inventory localStorage write ever failed (e.g. quota exceeded) during commit/release. Fixed: `useLocalInventory` now returns `saveError`, shown as a top-level banner in `App.jsx` (matching the same pattern designs already had).
  - Minor: `createDesign` now whitelists `isCommitted`/`committedBom` for defensive completeness, even though committing an unsaved draft is already blocked in the UI.
  - **Documented rather than engineered around** (deliberate scope calls, not oversights): no single cross-storage-key transaction between the designs write and the inventory write (each is individually atomic; not atomic together — see `feature-commit-inventory.md`); no cross-design stock reservation, so two different designs needing the same part can both show "buildable" until one actually commits (see `feature-stock-check.md`); no cross-tab staleness detection (`useLocalStorageState` doesn't listen for the `storage` event). All three are real, named limitations for a localStorage-only single-device app — building proper fixes (a combined transaction, a reservation system, or storage-event syncing) would be a disproportionate v1 investment versus the actual risk for a single staff member's normal workflow.
- Verified end-to-end in the browser via a real production-config build: BOM auto-calculation (including auto-derived gangway-hinge hardware from a gangway-attachment seam), stock shortfall display, commit deducting inventory exactly once (including under a forced double-click), release restocking exactly once, the full canvas/name/toggle/background lock while committed and unlock after release, and free-design mode showing the parts list without a stock check or Commit button — all with zero console errors.

## ❌ Tried But Failed
Nothing failed outright this session. The double-click race was initially going to be "fixed" by just disabling the button — testing (and the review's own reasoning) showed that alone doesn't close the gap without also making the underlying inventory update delta-based, since two absolute-value writes computed from the same stale snapshot converge to a wrong single value even without a UI race. Both fixes were needed together.
