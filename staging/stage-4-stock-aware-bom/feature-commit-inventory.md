# Feature — Commit to Inventory

Finalizing a design so its BOM is actually deducted from inventory — confirmed as a required v1 feature (both read-only check AND commit/deduct ship together, not staged). Inventory here is the same localStorage-backed state from Stage 1 — no database, no server (see `docs/master_plan.md`). **Done** — `DesignsPage.jsx`'s `handleCommit`/`handleRelease`, `useLocalInventory.js`'s `applyBulkDeltas`, `DesignEditor.jsx`'s lock-while-committed UI.

## How it works

- **Commit = deduct immediately** (resolved from the "reserve then deduct" open question) — simpler, and matches "committed" meaning "this design's parts have left inventory for a real job." A committed design records `isCommitted: true` and a `committedBom` snapshot (`[{partId, quantity}]`) of exactly what was deducted, so Release can restock precisely that, even if the catalog or the design's canvas changes afterward.
- **Release = full cancellation only** (resolved from the "full vs. partial release" open question) — adds the entire `committedBom` back to on-hand and clears the commit record. No partial release (e.g. "half these modules were actually installed") — simpler for v1, and the more realistic partial-build case is out of scope until it's shown to matter in practice.
- **A committed design is frozen** — canvas edits (add/drag/rotate/delete), the name field, the free-design toggle, and the background photo controls are all disabled until Released, so a committed design's canvas can never silently drift from what its `committedBom` actually deducted.
- **Atomicity within one storage key:** `applyBulkDeltas` in `useLocalInventory.js` computes each part's new quantity *inside* the `setQuantities` state updater (reading the current value at write time), not from a value the caller precomputed from a render-time snapshot — this is what makes commit/release correct even under a rapid double-click (verified by dispatching two synchronous clicks in the same tick and confirming inventory only moved once). `DesignEditor.jsx` also gates both actions behind a ref-based `isSubmitting` guard so a second click can't re-fire the handler at all while the first is still in flight.

## Known limitation (documented, not solved)

- **No single cross-key transaction.** A commit is two separate localStorage writes — one to `polydock:inventory:v1` (via `applyBulkDeltas`), one to `polydock:designs:v1` (via `saveDesign`, recording `isCommitted`/`committedBom`) — each individually atomic, but not atomic *together*. If the browser tab were killed between the two writes, inventory could be deducted while the design still shows `isCommitted: false` (or the reverse, if the order were swapped). Inventory is deducted first, then the design is marked committed, so an interruption leaves inventory already-deducted for a design that doesn't yet look committed — recoverable by noticing the mismatch and re-committing, rather than silently duplicating a deduction. A true single-transaction guarantee would need both to live under one storage key (or a real backend) — accepted as a rare-edge-case tradeoff for a localStorage-only app rather than engineered around now.
- **No cross-design reservation.** See `feature-stock-check.md`'s open question — two different designs both needing the same part can each show "buildable" until one of them actually commits.
- **Per-device, not shared** — a design "committed" on one device doesn't affect stock as seen on another device at all (same tradeoff noted throughout `docs/master_plan.md`), and even within one device, two browser tabs won't see each other's commits until reloaded (see `feature-stock-check.md`).
- **Deleting a committed design does not restock it** — the parts are treated as consumed by a real build, not reserved. `DesignsListPage.jsx` shows a stronger confirm warning when deleting a committed design specifically, since this is easy to expect to work the other way.
